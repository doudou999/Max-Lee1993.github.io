Git使用

发表于 2017-09-30 | 分类于 版本工具 | 阅读次数:

本文所讲的多有关于git的操作全部在window下，下面先开始讲一些准备工作。

mkdir test #在当前目录下创建一个文件夹test
cd test #进入test目录下
pwd #返回当前的文件文件路径

执行效果如下：

Administrator@YBKUIPCLJ0B14TA MINGW64 /c/Users/Administrator/Desktop
$ cd a

Administrator@YBKUIPCLJ0B14TA MINGW64 /c/Users/Administrator/Desktop/a
$ pwd
/c/Users/Administrator/Desktop/a

Administrator@YBKUIPCLJ0B14TA MINGW64 /c/Users/Administrator/Desktop/a
$

提交至仓库

1.初始化git

git init

执行该命令是在当前目录下创建一个空的仓库，显示为.git如果你没有看到.git文件夹，那是因为这个目录默认是隐藏的，用ls -ah命令就可以看见。

2.提交所有文件，当然可以指定要提交的文件git add test.txt

git add. #该命令是将项目中未被跟踪的所有文件都加入到仓库中。 它不提交这些文件，而只是让Git开始关注它们。

注：根据阮一峰老师博客的建议千万不要用window自带的文本编辑器编辑任何文件，否则会出很多麻烦。

3.用命令git commit告诉Git，把文件提交到仓库

git commit -m "提交一个一个更改" #此处双引号的是提交更改的提示内容，可按实际情况填写

1,2,3步可以总结为将文件提交至仓库，可以反复对文件进行这样的操作。注意在这个过程中可以使用git status查看仓库的当前状态，例如我某次的查看显示：

$ git status
On branch master
nothing to commit, working directory clean

说明工作区没有待提交之仓库的文件。

管理版本

当我们需要查看之前版本修改的记录，可以使用git log 命令查看，例如我某次的查看：

$ git log
commit 3c834c956d8a3b600ecb33c428db9fd86812c46a
Author: Max-Lee1993 <1208205420@qq.com>
Date:   Sat Sep 30 19:27:04 2017 +0800

starting over.

commit之后的一串字符串代表的是提交的id号。当然如果这个看起来比较乱的话，我们可以用git log --pretty=oneline来操作，结果如下：

$ git log --pretty=oneline
3c834c956d8a3b600ecb33c428db9fd86812c46a starting over.

当我们想回退到之前提交版本，可以使用git reset --hard commit_id，commit_id就是前面所说的一串字符串。例如：

$ git log
commit cd6ff9bf8bab5efb66039ce443eb2f730531a502
Author: Max-Lee1993 <1208205420@qq.com>
Date:   Sat Sep 30 20:07:30 2017 +0800

第二次提交.

commit ca0211c8540be188b8f8e8fb8a7ca476fbee689b
Author: Max-Lee1993 <1208205420@qq.com>
Date:   Sat Sep 30 20:06:16 2017 +0800

第一次提交.

从上面可以看出我有两次提交，退回到第一次提交：

$  git reset --hard ca0211
HEAD is now at ca0211c 第一次提交.

但是如果我想重新回到第二次提交怎么操作呢，使用git reflog查看命令历史，查看第二次提交的id号，例如:

$ git reflog
ca0211c HEAD@{0}: reset: moving to ca0211
cd6ff9b HEAD@{1}: commit: 第二次提交.
ca0211c HEAD@{2}: commit (initial): 第一次提交.

可以看到第二次提交的id号是cd6ff9b，然后使用git reset --hard cd6ff9b重新回到第二次提交，如下：

$ git reset --hard cd6ff9b
HEAD is now at cd6ff9b 第二次提交.

看下log，如下：

$ git log --pretty=oneline
cd6ff9bf8bab5efb66039ce443eb2f730531a502 第二次提交.
ca0211c8540be188b8f8e8fb8a7ca476fbee689b 第一次提交.

我们可以看到有两项提交，当前在第二次提交。当然这是向后退的办法，如果要向前，只需要git reset --hard HEAD^,意思是回到前一次的提交。不明白的朋友可以看阮一峰老师博客所讲的工作区和暂存区，相信会对上面的内容理解的更加透彻。

我们知道了工作区和暂存区以及提交区之后，下来还有一个命令：git checkout -- file，可以丢弃工作区的最近一次的修改。

当然如果在修改文件之后 ，放到暂存区了。但是又想撤销此次更改，使用git reset HEAD 112.txt命令可以将暂存区的修改回退到工作区。然后在工作区使用git checkout -- file

场景1：当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout – file。

场景2：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD file，就回到了场景1，第二步按场景1操作。

场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退，不过前提是没有推送到远程库。

以上引用来自阮一峰老师的博客。

用来删除工作区的文件，可以使用rm file命令，删除之后执行git status可以显示工作区和版本区的不同，这是可以commit删除的文件，版本区的file也会被删除，当然，如果没提交，可以使用git checkout -- file回到删之前的状态。

远程仓库

1.连接远程仓库（之前配置秘钥，这里以配置好秘钥来讲，有关秘钥配置请百度）

git remote add origin git@github.com:Max-Lee1993/test_git.git #这里以我的仓库为例，大家使用要改用户名和仓库名

2.本地仓库内容推送到远程仓库

git push -u origin master

如果出错，请参考下面的解决办法：

刚创建的github版本库，在push代码时出错：

$ git push -u origin master
To git@github.com:/Demo.git
! [rejected] master -master (non-fast-forward)
error: failed to push some refs to ‘git@github.com:/Demo.git’
hint: Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart. Merge the remote changes (e.g. ‘git pull’)
hint: before pushing again.
hint: See the ‘Note about fast-forwards’ in ‘git push –help’ for details.

网上搜索了下，是因为远程repository和我本地的repository冲突导致的，而我在创建版本库后，在github的版本库页面点击了创建README.md文件的按钮创建了说明文档，但是却没有pull到本地。这样就产生了版本冲突的问题。

有如下几种解决方法：

1.使用强制push的方法：

$ git push -u origin master -f

这样会使远程修改丢失，一般是不可取的，尤其是多人协作开发的时候。

2.push前先将远程repository修改pull下来

$ git pull origin master

$ git push -u origin master

从远程库克隆

使用以下命令：

git clone git@github.com:REMitchell/python-scraping.git

这是clone到本地了，如果你还有推送到远程仓库，那么需要执行：

git remote add origin git@github.com:Max-Lee1993/test_git.git #这里以我的仓库为例，大家使用要改用户名和仓库名

这里往往会出错，显示已经连接到远程仓库了，需要执行：

git remote rm origin

解除远程仓库的关联。然后在执行关联命令，之后就是push命令了。
